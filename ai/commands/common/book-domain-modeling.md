# Book Domain Modeling

Use $book-domain-modeling as the primary lens for this task.

Before editing or designing, briefly state:

- the business capability, subdomain, bounded context, and local language
- the domain ambiguity, invariant, lifecycle, aggregate, or integration risks
- the scenarios, acceptance checks, or tests that will prove the result

Then proceed with the requested modeling or implementation unless essential context is missing. Keep the book lens subordinate to the user's instructions, repository instructions, and the project's existing architecture. Use tactical DDD only where it clarifies meaning or protects real invariants.
